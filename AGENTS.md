## 行为准则（Karpathy Guidelines）

**权衡：这些准则偏向谨慎而非速度。对简单任务自行判断。**

### 1. 先想再写
- 明确陈述假设。不确定就问。
- 如果存在多种理解，列出它们——不要自己默默选一个。
- 如果更简单的方案可行，说出来。
- 如果有不清楚的地方，停下来，指出困惑点，问。

### 2. 简洁优先
**用最少代码解决问题。不加推测性东西。**
- 不实现没被要求的功能。
- 单次使用的代码不要抽象。
- 不处理不可能发生的错误。
- 写 200 行能做的事只用了 50 行就对了。
- 自问："高级工程师会说这过度复杂吗？" 如果是，简化。

### 3. 精准修改
**只碰该碰的。只清理你自己的烂摊子。**
- 不要"顺手改进"相邻代码、注释、格式。
- 不要重构没坏的东西。
- 匹配现有代码风格。
- 发现无关的死代码，提出来——不要直接删。
- 你的改动造成的孤立变量/导入——你来清理。
- 测试：每一行改动都能追溯到你的任务。

### 4. 目标驱动执行
**定义成功标准。循环直到验证通过。**
- "加验证" → "先写无效输入测试，再让它通过"
- "修 bug" → "先写复现测试，再修"
- 多步骤任务先列计划，每步标注验证方式。

## 项目目标

构建一个本地运行的 AI 视频创作工具：
- 左侧聊天框：调用 GPT 生成结构化视频脚本。
- 中间画布：展示和编辑镜头、三宫格故事板、视频片段。
- 底部时间线：像剪辑工具一样展示故事板轨和视频轨。
- 用户可以对单个图片或视频片段重生成并覆盖当前版本。
- 所有生成资产必须保存到本地项目目录。

## 技术栈

- React + Vite + TypeScript
- Electron
- Node.js + Fastify
- SQLite + Drizzle ORM
- Zustand
- Native Drag & Drop
- FFmpeg

## API 接入

- GPT 脚本：Packy OpenAI-compatible API
- 图片生成：Packy gpt-image-2 Images API
- 图生视频：RunningHub image-to-video API

## 代码规范

- 所有类型放在 packages/shared。
- API Key 只允许后端读取，不允许进入前端 bundle。
- 远程图片和视频结果必须下载到本地。
- 任务必须通过 jobs 表记录状态。
- 生成失败必须保留错误信息并支持重试。
- 不允许直接删除历史版本资产。
- 图片/视频多版本管理：每次重生成创建新文件不覆盖旧文件。
